## Technical Challenge: Build a Modular “System Topology Validator” in .NET 10

This repository contains my implementation of a compact but expressive engineering challenge designed to demonstrate clean architecture, modern .NET coding style, and senior‑level reasoning. The challenge is intentionally unusual to avoid typical tutorial patterns, while remaining small enough to complete in a single day.

The goal is to model a small fictional “system topology” and implement a modular validation engine that checks the topology against a set of well‑defined constraints.

---

### Domain Overview

The domain represents a simplified “system topology” consisting of **Nodes**, **Links**, and **Capabilities**. The validator checks whether the topology is structurally sound and whether nodes satisfy the constraints imposed by their capabilities and connections.

This domain is intentionally small but expressive enough to demonstrate architectural thinking, modelling, and rule‑based evaluation.

### Model Types

#### Node

Represents a component in the system.

```csharp
public sealed class Node
{
    public string Id { get; init; } = default!;
    public string Type { get; init; } = default!;
    public IReadOnlyDictionary<string, object> Attributes { get; init; } = default!;
    public IReadOnlyList<string> Capabilities { get; init; } = default!;
}
```

#### Link

Represents a connection between two nodes.

```csharp
public sealed class Link
{
    public string From { get; init; } = default!;
    public string To { get; init; } = default!;
    public string Kind { get; init; } = default!;
}
```

#### Topology

Represents the full system.

```csharp
public sealed class Topology
{
    public IReadOnlyList<Node> Nodes { get; init; } = default!;
    public IReadOnlyList<Link> Links { get; init; } = default!;
}
```

### Constraints

Each constraint is a pluggable rule implementing:

```csharp
public interface ITopologyRule
{
    RuleResult Evaluate(Topology topology);
}
```

Where:

```csharp
public sealed class RuleResult
{
    public bool Passed { get; init; }
    public string RuleName { get; init; } = default!;
    public IReadOnlyList<string> Messages { get; init; } = Array.Empty<string>();
}
```

#### Required Constraints

You must implement at least the following constraints:

---

#### 1. **No Orphan Nodes**

Every node must appear in at least one link (either as `From` or `To`).

**Failure example:**  
Node `cache-1` has no incoming or outgoing links.

---

#### 2. **No Cycles for “linear” Node Types**

Nodes of type `"linear"` must not form cycles.

**Example:**  
`A → B → C → A` is invalid if all three nodes are `"linear"`.

---

#### 3. **Capability Requires Attribute**

Some capabilities require specific attributes to exist.

Use this mapping:

| Capability | Required Attribute | Description |
|-----------|--------------------|-------------|
| `"storage"` | `"capacity"` | Storage nodes must define a capacity |
| `"compute"` | `"cores"` | Compute nodes must define number of cores |
| `"secure"` | `"encryption"` | Secure nodes must specify encryption type |

**Failure example:**  
Node `db-1` has capability `"storage"` but no `"capacity"` attribute.

---

#### 4. **Link Kind Compatibility**

Certain link kinds are only valid between certain node types.

Use this compatibility table:

| Link Kind | From Type | To Type |
|-----------|-----------|---------|
| `"data"` | `"compute"` | `"storage"` |
| `"control"` | `"compute"` | `"compute"` |
| `"replicate"` | `"storage"` | `"storage"` |

**Failure example:**  
A `"data"` link from a `"storage"` node to a `"compute"` node is invalid.

---

#### 5. **Max Fan‑Out for Compute Nodes**

Nodes of type `"compute"` may have at most **3 outgoing links**.

**Failure example:**  
Node `processor-1` has 5 outgoing links.
